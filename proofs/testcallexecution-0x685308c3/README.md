## Identification

All 33 function selectors in the deployed runtime resolve cleanly against the ABI of `TestCallExecution` in Piper Merriam's `pipermerriam/ethereum-alarm-canary` repo (and matching ABI in `pipermerriam/ethereum-alarm-client`):

- `wasSuccessful()` `reset()` `setBool()` `setUInt(uint256)` `setInt(int256)` `setAddress(address)` `setBytes32(bytes32)` `setBytes(bytes)` `setMany(uint256,int256,uint256,bytes20,address,bytes)`
- `doExecution(address)` `doLoops(uint256)`
- `registerBool/UInt/Int/Bytes/Bytes32/Address(address, value)` `registerData(address,int,bytes32,address)` `registerMany(address,uint,int,uint,bytes20,address,bytes)`
- `scheduleSetBool(address,uint256,bool)` `scheduleSetUInt(address,uint256,uint256)`
- plus 12 storage getters auto-generated by Solidity (`v_bool`, `v_uint`, `v_int`, `v_address`, `v_bytes32`, `v_bytes`, `vm_a`, `vm_b`, `vm_c`, `vm_d`, `vm_e`, `vm_f`)

The published `TestCallExecution` in tagged releases `v0.5.0`/`v0.6.0` of `pipermerriam/ethereum-alarm-clock` covers the `set*` / `do*` / register family. The `scheduleSet*` helpers are present in the on-chain bytecode but not in any tagged source release, so they were either in an untagged commit around the deployment date (Dec 2015) or merged in from the related `TestDataRegistry` contract by hand.

## Crack attempt result (NOT cracked)

Reconstructed source at `Testers_reconstructed.sol` merging:
- `TestCallExecution` from Testers.sol at v0.6.0 of pipermerriam/ethereum-alarm-clock
- `register*` family of functions (originally in a separate `TestDataRegistry` contract) merged into TestCallExecution since the on-chain bytecode has them in one contract
- Reconstructed `scheduleSetBool(address,uint256,bool)` and `scheduleSetUInt(address,uint256,uint256)` by reading the on-chain dispatch table and disassembling the function bodies

The schedule functions in on-chain bytecode call:
```
to.call.value(msg.value)(0x01991313, address(this), bytes4(sha3("setBool()" | "setUInt(uint256)")), blockNumber)
to.call(bytes4(sha3("registerData()")), value)
```
(0x01991313 is `keccak256("scheduleCall(address,bytes4,uint256)")[:4]`)

Diff result (soljson-v0.1.6 opt=1):
- mine 3,932 bytes, target 4,005 bytes (net -73 bytes; mine is missing code)
- First divergence at byte offset 40 (early in dispatch table)
- 5-byte insert at offset 392 (`6101009004`) suggests packed-bool storage shift
- 66-byte insert at offset 3,755: shared cleanup blocks for wasSuccessful = 0/1/2 writes
- Schedule function: target uses combined memory area for both calls (one buffer build); my version emits two independent call blocks

The compiler version (v0.1.6) is consistent with the deployment date (23 December 2015) and the canary build artifact's `compilerVersion: 0.1.6-d6e77ce0` field. The 73-byte deficit is in shared cleanup blocks and call-buffer layout that depends on subtle source structure not recoverable from the bytecode alone.

## Why it holds 4.9 ETH

The contract's `doLoops(uint iterations)` repeatedly sends 1 wei back to itself via `address(this).send(1)` to test the scheduler's ability to handle nested calls. Funds accumulated during test runs and were never withdrawn (the contract has no `withdraw` function).
